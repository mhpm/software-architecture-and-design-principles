# ¿Qué significa SRP?

## 📖 Definición

### 🇪🇸 Español

> **Una clase debe tener una sola razón para cambiar.**

Esto significa que una clase debe encargarse de **una única responsabilidad** dentro del sistema.

Si una clase realiza varias tareas distintas, cualquier cambio en una de esas tareas obligará a modificar la misma clase, aumentando el riesgo de introducir errores.

En otras palabras:

> **Una clase debe hacer una sola cosa y hacerla bien.**

---

### 🇺🇸 English

> **A class should have only one reason to change.**

A class should be responsible for only one part of the application's functionality.

If a class handles multiple responsibilities, changes to one responsibility may unintentionally affect the others.

In simple words:

> **A class should do one thing and do it well.**

---

# 🎯 Objetivo del principio

El objetivo del **Single Responsibility Principle (SRP)** es reducir la complejidad del software separando responsabilidades en clases independientes.

Cuando una clase tiene una sola responsabilidad:

- Es más fácil de entender.
- Es más fácil de modificar.
- Es más fácil de probar.
- Es más fácil reutilizarla.
- Es menos probable introducir errores al cambiarla.

---

# ¿Por qué existe este principio?

Imagina un edificio.

Cada departamento tiene una función:

- Recursos Humanos
- Finanzas
- Ventas
- Marketing

¿Qué pasaría si un solo departamento hiciera absolutamente todo?

Sería un desastre.

Lo mismo ocurre con las clases.

Cuando una clase hace demasiadas cosas:

- Es difícil de entender.
- Es difícil de probar.
- Es difícil de modificar.
- Es difícil encontrar errores.
- Es difícil reutilizarla.

Por eso nació SRP.

# 🌎 Ejemplo de la vida real

Imagina un restaurante.

Existe un empleado llamado José.

José:

- Cocina.
- Cobra.
- Lava los platos.
- Compra ingredientes.
- Atiende clientes.
- Lleva la contabilidad.

Al principio parece eficiente.

Pero después…

El cocinero pregunta:

> José, ¿ya compraste la carne?

El contador dice:

> José, necesito las facturas.

Un cliente pregunta:

> José, ¿puedes cobrarme?

Mientras tanto…

La comida se está quemando.

José no puede hacer todo.

Lo correcto sería:

👨‍🍳 **Cocinero**

Solo cocina.

💰 **Cajero**

Solo cobra.

🧹 **Personal de Limpieza**

Solo limpia.

📦 **Compras**

Solo compra.

Cada persona tiene **una única responsabilidad**.

# 💻 Ejemplo práctico en C#

Supongamos que estamos desarrollando **MultiplyNet**.

Necesitamos registrar miembros.

Un desarrollador escribe esto.

## ❌ Código Incorrecto

```csharp
public class MemberService
{
    public void Register(Member member)
    {
        Console.WriteLine("Registrando miembro...");
    }

    public void SendWelcomeEmail(Member member)
    {
        Console.WriteLine("Enviando correo...");
    }

    public void GenerateMembersReport()
    {
        Console.WriteLine("Generando reporte...");
    }

    public void PrintCredential(Member member)
    {
        Console.WriteLine("Imprimiendo credencial...");
    }

    public void ExportToExcel()
    {
        Console.WriteLine("Exportando Excel...");
    }

    public void BackupDatabase()
    {
        Console.WriteLine("Respaldando Base de Datos...");
    }
}
```

---

# 🤔 ¿Qué está mal?

Esta clase hace demasiadas cosas.

Además de registrar miembros:

- Envía correos.
- Genera reportes.
- Imprime credenciales.
- Exporta Excel.
- Hace respaldos.

Tiene muchas razones para cambiar.

Por ejemplo:

- Si cambia la forma de enviar correos...
- Si cambia el formato del reporte...
- Si cambia la exportación a Excel...
- Si cambia el sistema de respaldos...

Siempre tendremos que modificar la misma clase.

Esto viola el principio SRP.

---

# ✅ Código Correcto

```csharp
public class MemberService
{
    public void Register(Member member)
    {
        Console.WriteLine("Registrando miembro...");
    }
}
```

```csharp
public class EmailService
{
    public void SendWelcomeEmail(Member member)
    {
        Console.WriteLine("Correo enviado.");
    }
}
```

```csharp
public class ReportService
{
    public void GenerateMembersReport()
    {
        Console.WriteLine("Reporte generado.");
    }
}
```

```csharp
public class CredentialService
{
    public void Print(Member member)
    {
        Console.WriteLine("Credencial impresa.");
    }
}
```

```csharp
public class ExcelExporter
{
    public void Export()
    {
        Console.WriteLine("Excel generado.");
    }
}
```

```csharp
public class BackupService
{
    public void Backup()
    {
        Console.WriteLine("Backup realizado.");
    }
}
```

# 🧠 Explicación del código

Antes:

```text
1 clase
↓
6 responsabilidades
```

Después:

```text
6 clases
↓
1 responsabilidad por clase
```

Ahora el código es:

- Más limpio.
- Más reutilizable.
- Más fácil de probar.
- Más fácil de entender.
- Más fácil de mantener.

---

# 🚀 Beneficios de aplicar SRP

- ✅ Clases pequeñas y fáciles de leer.
- ✅ Menos errores al modificar el código.
- ✅ Código reutilizable.
- ✅ Pruebas unitarias más simples.
- ✅ Menor acoplamiento entre componentes.
- ✅ Facilita el trabajo en equipo.
- ✅ Hace que el sistema pueda crecer sin convertirse en un caos.

---

# ⚠️ Errores comunes

Evita crear clases llamadas:

- `Utils`
- `Helper`
- `Manager`
- `Common`
- `Service`

que terminan haciendo de todo.

Por ejemplo:

```text
SystemHelper.cs
```

con más de 5,000 líneas.

Eso suele ser una clara señal de que SRP se está violando.

---

# 💼 ¿Dónde se usa en proyectos reales?

En una aplicación ASP.NET Core es muy común encontrar clases como:

- `MemberService`
- `EmailService`
- `NotificationService`
- `ReportService`
- `ExportService`
- `AuditService`

Cada una tiene una única responsabilidad.

Este tipo de separación es habitual en aplicaciones empresariales y facilita el mantenimiento del sistema.
