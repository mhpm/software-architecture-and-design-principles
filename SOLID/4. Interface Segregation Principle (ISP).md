# 📚 Guía Definitiva de SOLID en C# y .NET

# Capítulo 4: I — Interface Segregation Principle (ISP)

> **"Ningún cliente debe verse obligado a depender de métodos que no utiliza."**
>
> — Robert C. Martin (Uncle Bob)

---

# ¿Qué significa ISP?

## 📖 Definición

### 🇪🇸 Español

> **Ninguna clase debe verse obligada a implementar métodos que no necesita.**

En otras palabras, **es mejor tener varias interfaces pequeñas y específicas que una interfaz enorme con muchas responsabilidades.**

Cada clase debe implementar únicamente los contratos que realmente necesita.

---

### 🇺🇸 English

> **Clients should not be forced to depend on methods they do not use.**

In other words, **it is better to have several small, focused interfaces than one large interface that forces unnecessary implementations.**

Each class should implement only the contracts it actually needs.

---

# 🎯 Objetivo del principio

El objetivo del **Interface Segregation Principle** es evitar que las clases implementen métodos innecesarios.

Cuando una interfaz tiene demasiadas responsabilidades:

- Se vuelve difícil de mantener.
- Obliga a implementar métodos vacíos.
- Produce excepciones como `NotImplementedException`.
- Aumenta el acoplamiento.
- Hace que el código sea más difícil de entender.

ISP propone dividir grandes interfaces en interfaces pequeñas y especializadas.

# 💻 Ejemplo práctico en C#

Supongamos que estamos desarrollando **MultiplyNet**.

Creamos esta interfaz.

## ❌ Código Incorrecto

```csharp
public interface IChurchWorker
{
    void Preach();

    void Sing();

    void PlayPiano();

    void ManageSound();

    void TeachChildren();
}
```

Ahora implementamos un ujier.

```csharp
public class Usher : IChurchWorker
{
    public void Preach()
    {
        throw new NotImplementedException();
    }

    public void Sing()
    {
        throw new NotImplementedException();
    }

    public void PlayPiano()
    {
        throw new NotImplementedException();
    }

    public void ManageSound()
    {
        throw new NotImplementedException();
    }

    public void TeachChildren()
    {
        throw new NotImplementedException();
    }
}
```

---

# 🤔 ¿Qué está mal?

Un ujier:

- No predica.
- No canta.
- No toca piano.
- No enseña niños.

Entonces...

¿Por qué implementa esos métodos?

Porque la interfaz fue mal diseñada.

---

# 🛠 Visualmente sería algo así

```text
          IChurchWorker
        ┌─────┼─────┐
        │     │     │
   Preach  Sing  Piano
        │
      Usher

❌ No necesita nada de eso
```

---

# ✅ Código Correcto

Dividimos la interfaz.

```csharp
public interface IPreacher
{
    void Preach();
}
```

```csharp
public interface ISinger
{
    void Sing();
}
```

```csharp
public interface IPianist
{
    void PlayPiano();
}
```

```csharp
public interface ISoundOperator
{
    void ManageSound();
}
```

```csharp
public interface IChildrenTeacher
{
    void TeachChildren();
}
```

Ahora cada ministerio implementa únicamente lo necesario.

Pastor.

```csharp
public class Pastor : IPreacher
{
    public void Preach()
    {
        Console.WriteLine("Predicando...");
    }
}
```

Director de Alabanza.

```csharp
public class WorshipLeader : ISinger
{
    public void Sing()
    {
        Console.WriteLine("Cantando...");
    }
}
```

Pianista.

```csharp
public class Pianist : IPianist
{
    public void PlayPiano()
    {
        Console.WriteLine("Tocando piano...");
    }
}
```

Multimedia.

```csharp
public class MultimediaOperator : ISoundOperator
{
    public void ManageSound()
    {
        Console.WriteLine("Operando sonido...");
    }
}
```

Ahora cada clase implementa únicamente lo que realmente necesita.

Cada clase implementa únicamente las necesarias.

# 🚀 Beneficios de aplicar ISP

- ✅ Interfaces fáciles de entender.
- ✅ Menor acoplamiento.
- ✅ Menos código innecesario.
- ✅ Evita `NotImplementedException`.
- ✅ Clases más limpias.
- ✅ Facilita las pruebas unitarias.
- ✅ Hace el sistema más flexible.

---

# ⚠️ Errores comunes

Una gran señal de alerta es encontrar esto:

```csharp
throw new NotImplementedException();
```

Dentro de una implementación.

Generalmente significa que la interfaz está obligando a implementar algo innecesario.

Otro síntoma:

Interfaces gigantes como:

```text
IApplicationService
```

Con:

- 40 métodos.
- 60 métodos.
- 100 métodos.

Eso normalmente viola ISP.
