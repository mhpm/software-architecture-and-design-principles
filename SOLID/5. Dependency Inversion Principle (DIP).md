# 📚 Guía Definitiva de SOLID en C# y .NET

# Capítulo 5: D — Dependency Inversion Principle (DIP)

> **"Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones. Las abstracciones no deben depender de los detalles. Los detalles deben depender de las abstracciones."**
>
> — Robert C. Martin (Uncle Bob)

---

# ¿Qué significa DIP?

## 📖 Definición

### 🇪🇸 Español

> **Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones.**

En otras palabras, una clase no debería depender directamente de otra clase concreta, sino de una **interfaz o abstracción**.

De esta forma, podemos cambiar la implementación sin afectar al resto del sistema.

---

### 🇺🇸 English

> **High-level modules should not depend on low-level modules. Both should depend on abstractions.**

In other words, classes should depend on **interfaces or abstractions instead of concrete implementations**.

This allows implementations to change without affecting the rest of the application.

---

# 🎯 Objetivo del principio

El objetivo del **Dependency Inversion Principle** es reducir el acoplamiento entre clases.

Cuando las clases dependen de abstracciones:

- Son más fáciles de reemplazar.
- Son más fáciles de probar.
- Son más reutilizables.
- Son más flexibles.
- Son más fáciles de mantener.

Este principio es la base de la **Inyección de Dependencias (Dependency Injection)** utilizada en ASP.NET Core.

---

# ¿Por qué existe este principio?

Imagina que compras una lámpara.

La lámpara necesita un foco.

Pero...

¿Depende de un foco Philips?

No.

¿Depende de un foco GE?

No.

¿Depende de un foco Osram?

Tampoco.

La lámpara solamente depende de algo que cumpla el contrato de:

> "Ser un foco compatible."

Mañana puedes cambiar el foco.

La lámpara sigue funcionando.

Eso es exactamente DIP.

---

# 🌎 Ejemplo de la vida real

Piensa en un automóvil.

El automóvil necesita combustible.

Pero no depende de una gasolinera específica.

Puedes cargar combustible en:

- Shell
- BP
- Mobil
- Pemex

Mientras todas proporcionen combustible compatible, el automóvil funciona exactamente igual.

El automóvil depende del combustible, no de una empresa específica.

---

# ⛪ Ejemplo aplicado a una Iglesia

Imagina que el pastor dice:

> "Solo Juan puede dirigir la alabanza."

¿Qué ocurre si Juan se enferma?

No hay alabanza.

Todo depende de una persona.

Ahora imagina otra iglesia.

El pastor dice:

> "Necesitamos un Líder de Alabanza."

Puede ser:

- Juan
- Pedro
- María
- Ana

Mientras cualquiera pueda desempeñar el rol correctamente, la iglesia sigue funcionando.

La iglesia depende del **cargo**, no de la persona.

---

# ⭐ Principio importante

> **Depende del rol, no de la persona.**

En programación:

> **Depende de interfaces, no de clases concretas.**

---

# 💻 Ejemplo práctico en C#

Supongamos que estamos desarrollando **MultiplyNet**.

Queremos enviar notificaciones.

## ❌ Código Incorrecto

```csharp
public class EmailService
{
    public void Send(string message)
    {
        Console.WriteLine($"Email: {message}");
    }
}
```

```csharp
public class NotificationService
{
    private readonly EmailService _emailService;

    public NotificationService()
    {
        _emailService = new EmailService();
    }

    public void Notify(string message)
    {
        _emailService.Send(message);
    }
}
```

---

# 🤔 ¿Qué está mal?

`NotificationService` depende directamente de `EmailService`.

¿Qué pasa si mañana queremos enviar:

- SMS
- WhatsApp
- Telegram
- Push Notification

Debemos modificar `NotificationService`.

Está completamente acoplado a una implementación específica.

---

# 🛠 Visualmente sería algo así

```text
NotificationService
        │
        ▼
  EmailService
```

No existe flexibilidad.

---

# ✅ Código Correcto

Creamos una abstracción.

```csharp
public interface IMessageService
{
    void Send(string message);
}
```

---

Creamos una implementación.

```csharp
public class EmailService : IMessageService
{
    public void Send(string message)
    {
        Console.WriteLine($"Email: {message}");
    }
}
```

Otra implementación.

```csharp
public class SmsService : IMessageService
{
    public void Send(string message)
    {
        Console.WriteLine($"SMS: {message}");
    }
}
```

Otra más.

```csharp
public class WhatsAppService : IMessageService
{
    public void Send(string message)
    {
        Console.WriteLine($"WhatsApp: {message}");
    }
}
```

Ahora el servicio principal.

```csharp
public class NotificationService
{
    private readonly IMessageService _messageService;

    public NotificationService(IMessageService messageService)
    {
        _messageService = messageService;
    }

    public void Notify(string message)
    {
        _messageService.Send(message);
    }
}
```

---

Podemos usar cualquier implementación.

```csharp
var notification = new NotificationService(new EmailService());

notification.Notify("Bienvenido");
```

O cambiarla por otra.

```csharp
var notification = new NotificationService(new SmsService());

notification.Notify("Bienvenido");
```

O incluso.

```csharp
var notification = new NotificationService(new WhatsAppService());

notification.Notify("Bienvenido");
```

Nunca modificamos `NotificationService`.

---

# 🌟 ¿Cómo funciona esto en ASP.NET Core?

Aquí es donde realmente cobra vida este principio.

Registramos una implementación.

```csharp
builder.Services.AddScoped<IMessageService, EmailService>();
```

Luego simplemente la inyectamos.

```csharp
public class NotificationService
{
    private readonly IMessageService _messageService;

    public NotificationService(IMessageService messageService)
    {
        _messageService = messageService;
    }

    public void Notify(string message)
    {
        _messageService.Send(message);
    }
}
```

Si mañana queremos usar SMS.

Solo cambiamos:

```csharp
builder.Services.AddScoped<IMessageService, SmsService>();
```

¡No modificamos una sola línea de `NotificationService`!

---

# 🧠 Explicación del código

Antes.

```text
NotificationService
        │
        ▼
 EmailService
```

Dependencia rígida.

Después.

```text
           IMessageService
                 ▲
      ┌──────────┼──────────┐
      │          │          │
 EmailService SmsService WhatsAppService
                 ▲
                 │
       NotificationService
```

Ahora el sistema depende del contrato (`IMessageService`), no de una implementación específica.

# 🚀 Beneficios de aplicar DIP

- ✅ Bajo acoplamiento.
- ✅ Código más flexible.
- ✅ Fácil de probar (Mocks).
- ✅ Fácil de extender.
- ✅ Facilita la Inyección de Dependencias.
- ✅ Implementaciones intercambiables.
- ✅ Arquitecturas limpias y mantenibles.

---

# ⚠️ Errores comunes

Una gran señal de alerta es encontrar código como este.

```csharp
var service = new EmailService();
```

O.

```csharp
private readonly EmailService _service;
```

Dentro de otra clase.

Eso indica que la clase depende directamente de una implementación concreta.

En la mayoría de los casos debería depender de una interfaz.

## ¿Qué es el Dependency Inversion Principle?

### 🇪🇸 Respuesta

El **Dependency Inversion Principle (DIP)** establece que los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones. Esto reduce el acoplamiento, facilita el mantenimiento y permite intercambiar implementaciones sin modificar el código de negocio.

### 🇺🇸 Answer

---

# 💡 Relación entre DIP y Dependency Injection

Es muy común confundir estos conceptos, pero **no son lo mismo**.

- **DIP (Dependency Inversion Principle)** es un **principio de diseño**.
- **Dependency Injection (DI)** es una **técnica para implementar ese principio**.

En ASP.NET Core, el contenedor de servicios (`builder.Services`) implementa la Inyección de Dependencias, permitiéndonos aplicar DIP de forma sencilla.
