# Capítulo 2: O — Open/Closed Principle (OCP)

> **"Las entidades de software deben estar abiertas para extensión, pero cerradas para modificación."**
>
> — Robert C. Martin (Uncle Bob)

---

# ¿Qué significa OCP?

## 📖 Definición

### 🇪🇸 Español

> **Las entidades de software (clases, módulos, funciones, etc.) deben estar abiertas para extensión, pero cerradas para modificación.**

Esto significa que debemos poder **agregar nuevas funcionalidades sin modificar el código existente**.

En otras palabras, cuando el negocio cambia o aparecen nuevos requerimientos, no deberíamos tener que editar clases que ya funcionan correctamente. En lugar de eso, deberíamos **extender el comportamiento** mediante nuevas implementaciones.

---

### 🇺🇸 English

> **Software entities should be open for extension but closed for modification.**

This means that software should allow **new functionality to be added without changing existing code**.

Instead of modifying code that already works, we should extend the system by creating new implementations.

---

# 🎯 Objetivo del principio

El objetivo del **Open/Closed Principle** es evitar que el software se vuelva frágil.

Cada vez que modificamos código existente existe el riesgo de:

- Introducir nuevos errores.
- Romper funcionalidades existentes.
- Afectar otros módulos.
- Incrementar el costo de mantenimiento.

Con OCP buscamos que el sistema **crezca sin romper lo que ya funciona**.

---

# ¿Por qué existe este principio?

Imagina un edificio de oficinas.

Cada vez que una empresa nueva renta un piso, ¿sería necesario demoler el edificio para agregar una oficina?

Por supuesto que no.

Simplemente se habilita un nuevo espacio.

El edificio fue diseñado para crecer.

Eso mismo busca OCP.

# 💻 Ejemplo práctico en C#

Supongamos que estamos desarrollando **MultiplyNet**.

Necesitamos enviar notificaciones.

Inicialmente solo existe Email.

## ❌ Código Incorrecto

```csharp
public class NotificationService
{
    public void Send(string type, string message)
    {
        if (type == "Email")
        {
            Console.WriteLine($"Enviando Email: {message}");
        }
        else if (type == "SMS")
        {
            Console.WriteLine($"Enviando SMS: {message}");
        }
    }
}
```

---

# 🤔 ¿Qué está mal?

Hoy funciona.

Pero mañana el cliente dice:

> Ahora quiero enviar WhatsApp.

Entonces modificamos la clase.

Después pide:

- Telegram
- Push Notification
- Discord
- Slack

Y cada vez debemos regresar a modificar la misma clase.

Cada modificación aumenta el riesgo de romper algo que ya funcionaba.

Eso viola OCP.

---

# 🛠 Visualmente sería algo así

```text
              NotificationService
                     │
        ┌────────────┼─────────────┐
        │            │             │
      Email         SMS      WhatsApp
                        ↑
             Siempre modificando
```

Cada nueva funcionalidad implica editar la misma clase.

---

# ✅ Código Correcto

Primero definimos un contrato.

```csharp
public interface INotification
{
    void Send(string message);
}
```

---

Creamos una implementación para Email.

```csharp
public class EmailNotification : INotification
{
    public void Send(string message)
    {
        Console.WriteLine($"Enviando Email: {message}");
    }
}
```

---

Otra para SMS.

```csharp
public class SmsNotification : INotification
{
    public void Send(string message)
    {
        Console.WriteLine($"Enviando SMS: {message}");
    }
}
```

---

Otra para WhatsApp.

```csharp
public class WhatsAppNotification : INotification
{
    public void Send(string message)
    {
        Console.WriteLine($"Enviando WhatsApp: {message}");
    }
}
```

---

El servicio queda así.

```csharp
public class NotificationService
{
    public void Notify(INotification notification, string message)
    {
        notification.Send(message);
    }
}
```

---

Ahora podemos usar cualquier implementación.

```csharp
var service = new NotificationService();

service.Notify(new EmailNotification(), "Bienvenido.");

service.Notify(new SmsNotification(), "Bienvenido.");

service.Notify(new WhatsAppNotification(), "Bienvenido.");
```

Nunca volvemos a modificar `NotificationService`.

Solo agregamos nuevas clases.

---

# 🧠 Explicación del código

Antes:

```text
NotificationService
        │
     if Email
     if SMS
     if WhatsApp
     if Telegram
     if Push
     if ...
```

Cada nueva opción implicaba modificar la clase.

Después:

```text
NotificationService
        │
        ▼
 INotification
      │
 ┌────┼────┬──────────┐
 │    │    │          │
Email SMS WhatsApp Telegram
```

El sistema ahora crece agregando nuevas implementaciones.

# 🚀 Beneficios de aplicar OCP

- ✅ Menor riesgo de introducir errores.
- ✅ Código más estable.
- ✅ Fácil de extender.
- ✅ Más reutilizable.
- ✅ Facilita el mantenimiento.
- ✅ Permite agregar nuevas funcionalidades rápidamente.
- ✅ Favorece arquitecturas escalables.

---

# ⚠️ Errores comunes

Una señal de que estás violando OCP es encontrar código como este:

```csharp
if(type == "A")
{
}
else if(type == "B")
{
}
else if(type == "C")
{
}
else if(type == "D")
{
}
```

O grandes bloques de:

```csharp
switch(type)
{
    ...
}
```

Cada nuevo caso obliga a modificar el mismo archivo.

Muchas veces eso indica que deberías usar **polimorfismo** mediante interfaces o clases base.
