# 📚 Guía Definitiva de SOLID en C# y .NET

# Capítulo 3: L — Liskov Substitution Principle (LSP)

> **"Los objetos de una clase derivada deben poder sustituir a los objetos de la clase base sin alterar el correcto funcionamiento del programa."**
>
> — Barbara Liskov (1987)

---

# ¿Qué significa LSP?

## 📖 Definición

### 🇪🇸 Español

> **Los objetos de una clase derivada deben poder sustituir a los objetos de su clase base sin alterar el correcto funcionamiento del programa.**

Esto significa que **si una clase hereda de otra, debe comportarse como la clase padre espera**.

Una clase hija nunca debe romper las reglas o el comportamiento prometido por su clase base.

---

### 🇺🇸 English

> **Objects of a superclass should be replaceable with objects of its subclasses without affecting the correctness of the program.**

This means that **derived classes must honor the behavior expected from their base class.**

A subclass should never surprise or break the code that uses the base class.

---

# 🎯 Objetivo del principio

El objetivo del **Liskov Substitution Principle** es garantizar que el polimorfismo funcione correctamente.

Cuando una clase hereda de otra, cualquier lugar que espere la clase padre debe poder recibir la clase hija sin producir errores ni comportamientos inesperados.

---

# ¿Por qué existe este principio?

Imagina que compras un automóvil.

El vendedor dice:

> "Todos nuestros automóviles pueden conducir."

Entonces compras uno.

Pero resulta que ese automóvil solamente puede avanzar en reversa.

¿Cumplió con la promesa de ser un automóvil?

No.

Aunque técnicamente sea un automóvil, no cumple las expectativas.

Eso mismo ocurre cuando una clase hija rompe el comportamiento esperado de la clase padre.

---

No.

No.

# ⭐ Principio importante

> **Si alguien ocupa un rol, debe comportarse de acuerdo con ese rol.**

En programación:

> **Si una clase hereda de otra, debe cumplir el comportamiento esperado por esa clase.**

---

# 💻 Ejemplo práctico en C#

Supongamos que estamos desarrollando **MultiplyNet**.

Queremos representar líderes.

## ❌ Código Incorrecto

```csharp
public class Leader
{
    public virtual void LeadMeeting()
    {
        Console.WriteLine("Dirigiendo reunión.");
    }
}
```

Creamos otra clase.

```csharp
public class Visitor : Leader
{
    public override void LeadMeeting()
    {
        throw new NotImplementedException();
    }
}
```

Ahora alguien escribe:

```csharp
Leader leader = new Visitor();

leader.LeadMeeting();
```

Resultado:

```
NotImplementedException
```

El sistema falla.

---

# 🤔 ¿Qué está mal?

La clase `Visitor` hereda de `Leader`.

Por lo tanto, cualquier lugar que espere un `Leader` debería poder recibir un `Visitor`.

Pero no ocurre.

El visitante rompe el comportamiento esperado.

Por eso esta herencia es incorrecta.

---

# 🛠 Visualmente sería algo así

```text
              Leader
                 ▲
                 │
             Visitor

¿Puede dirigir una reunión?

❌ No
```

La herencia no tiene sentido.

---

# ✅ Código Correcto

En lugar de heredar incorrectamente, definimos un contrato.

```csharp
public interface ILeader
{
    void LeadMeeting();
}
```

Creamos implementaciones válidas.

```csharp
public class CellLeader : ILeader
{
    public void LeadMeeting()
    {
        Console.WriteLine("Dirigiendo célula.");
    }
}
```

```csharp
public class YouthLeader : ILeader
{
    public void LeadMeeting()
    {
        Console.WriteLine("Dirigiendo reunión de jóvenes.");
    }
}
```

Ahora cualquier líder funciona correctamente.

```csharp
List<ILeader> leaders =
[
    new CellLeader(),
    new YouthLeader()
];

foreach(var leader in leaders)
{
    leader.LeadMeeting();
}
```

Todos cumplen el contrato.

# 🧠 Otro ejemplo clásico

## ❌ Incorrecto

```csharp
public class Bird
{
    public virtual void Fly()
    {
        Console.WriteLine("Volando...");
    }
}
```

```csharp
public class Penguin : Bird
{
    public override void Fly()
    {
        throw new NotImplementedException();
    }
}
```

Problema:

Los pingüinos no vuelan.

Entonces nunca debieron heredar de una clase que obliga a volar.

---

## ✅ Correcto

```csharp
public abstract class Bird
{
}
```

```csharp
public interface IFlyable
{
    void Fly();
}
```

```csharp
public class Eagle : Bird, IFlyable
{
    public void Fly()
    {
        Console.WriteLine("Volando...");
    }
}
```

```csharp
public class Penguin : Bird
{
}
```

Ahora cada clase representa correctamente la realidad.

# 🧠 Explicación del código

Antes:

```text

   ▲
Visitor

Visitor rompe el comportamiento.
```

Después:

```text
          ILeader
             ▲
      ┌──────┴──────┐
      │             │
CellLeader   YouthLeader
```

Ambos cumplen exactamente el mismo contrato.

# 🚀 Beneficios de aplicar LSP

- ✅ Herencias correctas.
- ✅ Polimorfismo confiable.
- ✅ Menos errores en tiempo de ejecución.
- ✅ Código más predecible.
- ✅ Facilita el mantenimiento.
- ✅ Evita excepciones inesperadas.

---

# ⚠️ Errores comunes

Señales de que probablemente estás violando LSP:

- Sobrescribir un método solo para lanzar una excepción.
- Sobrescribir un método dejando el cuerpo vacío.
- Heredar únicamente para reutilizar código.
- Crear clases hijas que no cumplen las expectativas del padre.

Ejemplo:

```csharp
throw new NotImplementedException();
```

Dentro de un método heredado.

Eso suele ser una gran señal de alerta.

# 🎤 Pregunta típica de entrevista

## ¿Qué es el Liskov Substitution Principle?

### 🇪🇸 Respuesta

El **Liskov Substitution Principle (LSP)** establece que cualquier objeto de una clase hija debe poder reemplazar a un objeto de la clase padre sin alterar el comportamiento esperado del sistema. Si una clase derivada rompe ese comportamiento, entonces la herencia está mal diseñada.

### 🇺🇸 Answer

The **Liskov Substitution Principle (LSP)** states that objects of derived classes should be replaceable with objects of their base class without affecting the correctness of the application. If a subclass breaks the expected behavior, the inheritance hierarchy is incorrect.
